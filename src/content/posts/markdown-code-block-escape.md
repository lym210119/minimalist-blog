## 你遇到的现象

假设你问一个 AI：“请展示一段 Markdown 代码，里面包含一个 bash 代码块。”

AI 可能回复：

```markdown
这是一个示例：
```bash
echo "hello"
```
```

但是，在你看到的渲染结果中，**代码块的内容并没有被完整包在代码框里**，而是像这样：

> 这是一个示例：
>
> `bash`  
> echo “hello”  
> ` `

甚至后面的文本直接裸露在外面，完全不是预期的样子。

**这就是“被渲染出代码块外面”**——本该被包裹在代码块里的内容，因为 Markdown 解析器的误判，被当成了普通文本。

## 为什么会这样？

根本原因：**外层代码块的反引号数量与内部一样多，导致解析器提前关闭了代码块**。

Markdown 解析器在遇到 ` ``` ` 时，会认为一个代码块开始了。当它再次遇到同样数量的 ` ``` ` 时，就认为代码块结束了。

看看 AI 刚才输出的原始文本：

```text
```markdown
这是一个示例：
```bash
echo "hello"
```
```
```

解析器的工作流程：

1. 读到第一个 ` ``` ` → 代码块开始，语言标记为 `markdown`。
2. 继续读：`这是一个示例：\n`（普通文本，属于代码块内容）。
3. 然后读到 ` ```bash ` → 解析器看到 ` ``` `，**认为这是当前代码块的结束标记**。
4. 于是代码块在 `这是一个示例：` 之后就结束了，后面的 `echo "hello"` 和 ` ``` ` 都变成了代码块**外面**的普通文本。
5. 最终渲染效果就是混乱的。

## 核心规则：外层反引号数量必须大于内部

要解决这个问题，**外层代码块使用的反引号数量，必须严格大于内部任何一处连续反引号的数量**。

因为内部需要展示标准的三个反引号（例如 ` ```bash `），所以外层至少使用 **4 个反引号**。

### 正确的输出格式（AI 应该这样回复）

AI 应该输出以下原始文本（注意外层是 **4 个反引号**）：

````
````markdown
这是一个示例：
```bash
echo "hello"
```
````
````

当你的 Markdown 渲染器看到 **4 个反引号** 开始的代码块时，它会寻找 **4 个反引号** 作为结束标记。内部的 ` ``` ` 因为数量不同（只有 3 个），不会被误认为是结束标记。

这样，整个内容就会被完整地包裹在一个代码块中，你看到的效果就是一个干净的代码框，里面完整展示了一段包含嵌套代码块的 Markdown 源码。

## 实战对比

### ❌ 错误示范（AI 用三个反引号）

```text
```markdown
```bash
echo "hello"
```
```
```

渲染结果：内容被截断，部分裸露在外面。

### ✅ 正确示范（AI 用四个反引号）

```text
````markdown
```bash
echo "hello"
```
````
```

渲染结果：完美展示一个嵌套代码块。

## 为什么 AI 特别容易犯这个错误？

- AI 在训练时，看到的普通 Markdown 文档大多使用三个反引号。
- AI 并不知道自己正在一个“会被 Markdown 渲染”的环境中输出。
- 如果 AI 不加思考地直接输出三个反引号包裹的、内部又包含三个反引号的文本，就必然导致渲染错乱。

因此，**AI 开发者需要在提示词或后处理中，让模型学会在展示 Markdown 示例时自动升级为四个反引号**。

## 总结

| 你看到的现象 | 根本原因 | 解决方案 |
| --- | --- | --- |
| AI 输出的代码块内容跑到了框外面 | 外层用了 3 个反引号，内部也用了 3 个，导致解析器提前关闭代码块 | 外层使用 **4 个或更多** 反引号 |

下次你看到 AI 的回复中代码块“炸”了，就知道它不是乱码，而是忘记用四个反引号了。如果你是自己写提示词，记得加上一句：

> “当你需要展示包含三个反引号代码块的 Markdown 源码时，请用四个反引号包裹外层。”
